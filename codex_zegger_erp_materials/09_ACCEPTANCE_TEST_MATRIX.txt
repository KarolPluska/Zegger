ZEGGER ERP - ACCEPTANCE TEST MATRIX
FINAL QA / UAT / REGRESSION / MIGRATION / SECURITY / UX / MOBILE CHECKLIST

PURPOSE
This file defines the acceptance test matrix for the full Zegger ERP implementation.
It is intended for Codex as a non-optional completion target.
The implementation is NOT accepted until the items below are satisfied or explicitly documented as not applicable with a technical reason.

CORE RULES
1. No regression in the legacy Offer Panel user experience, business flow, PDF generation, customer handling, and offer history.
2. New ERP shell, modules, permissions, communicator, company model, relations, sources, notifications, and calculator integration must work as a coherent system.
3. Any deviation from the matrix must be documented in delivery notes.
4. For every failed test there must be: root cause, fix, and re-test result.
5. All tests must be validated on desktop and mobile where applicable.
6. Browser minimum manual coverage: Chrome, Edge, Safari.
7. Device coverage minimum: desktop, tablet, mobile.
8. OS coverage minimum: Windows and macOS for desktop behavior.
9. For all destructive actions there must be a confirmation flow or an intentional documented UX choice.
10. For all permission boundaries there must be backend validation, not only frontend hiding.

TEST EXECUTION FORMAT
For each test case, Codex should be able to state:
- STATUS: PASS / FAIL / BLOCKED / N-A
- BUILD/VERSION TESTED
- PRECONDITIONS
- STEPS EXECUTED
- EXPECTED RESULT
- ACTUAL RESULT
- NOTES / SCREEN / LOG REFERENCE
